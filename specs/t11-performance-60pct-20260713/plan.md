# T11 六用例性能优化计划

## 基线与证据

1. 固定工作树、代码提交、Python/GDAL/GeoPandas 环境和冻结 T10 六用例输入。
2. 按固定顺序运行 T11 提取，记录入口墙钟耗时、T11 summary 耗时、最大常驻内存和退出状态。
3. 对 `1885118` 生成函数级 profile，识别累计耗时和调用次数热点。
4. 保存优化前结构化输出，作为业务等价金标准。

## 实现策略

- 只实施由 profile 支持、且不改变候选选择与输出语义的优化。
- 优先消除重复 I/O、重复解析、重复空间索引构建和可证明等价的重复遍历。
- 在既有入口中以批量 Case 编排摊薄依赖启动成本，并用受控并行提高相互独立 Case 的吞吐；保留单用例调用兼容。
- 不减少输出文件，不改变排序、字段、几何、CRS 或审计内容。
- 如单文件接近 60 KiB，先做契约清晰的内部拆分，再实施优化。

## 验证顺序

1. T11 定向单元测试。
2. `1885118` 优化后回归与结构化输出等价比较。
3. 通过正式批量入口按固定 Case 顺序完整回归，允许内部受控并行，但 stdout 与输出目录保持 Case 顺序和隔离。
4. 汇总总耗时、逐用例耗时、峰值内存和业务等价结论。
5. T11 人工复核相关兼容性测试、源码体量审计、仓库状态审计。

## 性能计算

- 基线总耗时：六次独立 T11 提取进程的入口墙钟耗时之和。
- 优化总耗时：同一正式入口批量模式覆盖相同六用例的单次入口墙钟耗时。
- 耗时比：`optimized_total / baseline_total`。
- 吞吐提升：`baseline_total / optimized_total - 1`。
- 目标：耗时比不高于 `0.60`。

## 风险控制

- 首次运行的文件系统缓存可能影响结果：保留优化前逐用例和优化后批量入口原始值，并固定输入、环境和 Case 顺序。
- 并行会提高峰值内存：限制 `workers` 为 `1..8`，默认 `2`，并记录正式回归峰值 RSS。
- GeoPackage/XLSX 写入可能占主要耗时：不得通过少写或降级格式优化。
- 空间索引或向量化改写可能改变边界判定：必须以几何及候选结果逐项等价为门禁。
- 性能若未达标，继续回到 profile 证据定位，不放宽业务等价口径。
