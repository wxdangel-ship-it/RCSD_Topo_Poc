# default-imp

## 适用范围

- 默认用于非大型 Spec Kit 任务。
- 默认用于小中型编程修改。
- 不适用于需求新增、跨模块重构、边界不清的大型任务。

## 目标

- 先修订，再执行。
- 降低脑补需求、顺手重构、扩大改动范围、误删无关代码等风险。
- 在不增加用户重复说明成本的前提下，提供统一默认执行方式。

## 轻量流程

### 第一步：先修订

快速明确：

- 当前目标是什么。
- 本轮改动边界是什么。
- 是否存在关键不确定项。
- 本轮准备如何验证。

### 第二步：判断是否可直接执行

- 如果目标清晰、改动范围有限、风险可控，则直接执行。
- 如果边界不清、影响扩大、业务口径不稳，则先回报“修订卡”，不要直接写代码。

修订卡至少说明：

- 当前目标理解。
- 暂定改动边界。
- 关键不确定项。
- 建议先确认的问题。

### 第三步：实现

- 只做最小必要改动。
- 不顺手重构。
- 不做额外抽象。
- 不修改与当前任务无关的代码、注释、文档、命名和结构。
- 不删除未完全理解的历史逻辑或兼容代码。

### 第四步：验证

完成后区分：

- 已修改
- 已验证
- 待确认

不要把“看起来应该可以”表述成“已经修复”。

### 第五步：回报

完成后至少回报：

- 修改了哪些文件。
- 每个文件修改目的。
- 如何验证。
- 当前结果属于：已修改 / 已验证 / 待确认。

如需 GPT 继续判断，仅整理必要材料到指定目录或网络硬盘对应目录，并明确：

- 目录路径。
- 建议优先阅读文件。
- 希望 GPT 判断的问题。

## 默认实现守则

- 不脑补需求。
- 只做当前目标的最小必要改动。
- 不顺手重构无关代码。
- 不修改与当前任务无关的注释、文档、命名和结构。
- 不删除未完全理解的历史逻辑或兼容代码。
- 优先简单、可验证、可回退的实现。
- 结果要区分“已修改 / 已验证 / 待确认”。

## 与 Spec Kit 的关系

- `default-imp` 不是 Spec Kit 的替代品。
- 大任务仍走 Spec Kit。
- 小任务走 `default-imp`。
- 若小任务执行中发现范围扩大、边界不清、需求不稳，应升级为 Spec Kit 流程。
- 大任务的 implement 阶段，也可以引用该 Skill 作为轻量实现守则。
