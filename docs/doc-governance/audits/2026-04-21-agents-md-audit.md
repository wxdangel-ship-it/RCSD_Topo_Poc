## 0. 执行摘要

当前 `AGENTS.md` 是 24 条平铺要点 + 7 条"轻量编码守则"，结构问题主要有四类：

1. **"硬规则、软建议、信息条目"混在同一个无层级列表里**，大模型逐条扫描时所有条目权重接近，关键约束容易被淹没。
2. **关键约束的措辞动词偏弱**（"视为"、"应"、"建议"），缺少"必须 / 禁止 / 停机"这类可触发模型 stop-and-confirm 行为的强信号。
3. **几条核心规则只有"事后触发"语义，没有"事前阻止"语义**，文件体量规则就是典型——只有在"必须触碰超阈值文件"时才被激活，对"在普通文件里随手堆代码导致它逼近 100 KB"完全无约束。
4. **部分规则的真相已经过期或与代码事实脱钩**，但 `AGENTS.md` 仍把它们当成可信来源，反过来误导后续轮次。

本审计建议：

- 重写为**分组 + 优先级 + 动词强化**的结构（草稿见同目录另一文件）。
- 将"代码体量"从一条软规则升级为**带前置检查 + 硬停机 + CI 闭环**的三段式硬约束。
- 把"项目内只有 4 条具体业务模块"等会过期的事实从 `AGENTS.md` 中移走，改为只引用 `module-lifecycle.md`。
- 删除与 `.agents/skills/default-imp/SKILL.md` 重复的"CodeX 默认轻量编码守则"段，避免双源真相。

## 2. 失控点深度分析

### 2.1 文件体量规则（line 18，最突出的失效点）

**原文**：

> 文件体量：单个源码 / 脚本文件超过 `100 KB` 视为结构债；后续变更若必须触碰超阈值文件，先给出拆分计划或结构整改说明；如本轮只允许治理修文，也必须至少给出豁免说明或引用现有拆分计划。

**问题分解**：

1. **触发条件是"事后 + 必须触碰"**：模型理解为"我没主动改这个文件就不用管"，但现实里"在原文件里加 200 行新代码"也是触碰，规则没有把"增量长度阈值"或"行数阈值"显式化。
2. **"视为结构债"是描述句，不是命令句**：缺少"必须停机 / 必须报告 / 禁止继续追加"这种动词，模型不会触发停机。
3. **"豁免说明"是无成本逃生口**：只要写一句"本轮只是修 bug，暂不拆分"就满足条件。审计了实际执行，多轮里都用过这种说法。
4. **没有"新文件出生即 100 KB"的拦截**：`stage4_step4_event_interpretation.py` (118.6 KB) 与 `stage4_geometry_utils.py` (104.5 KB) 都是这一周内新增到超阈值的文件，没有任何审计记录。
5. **没有反馈闭环**：`docs/repository-metadata/code-size-audit.md` 是手工维护的快照（最近一次 2026-04-14），但 `AGENTS.md` 没有要求"任何超阈值变更必须刷新该表"，导致表与代码事实脱钩。

**实测证据（2026-04-21 现场盘点）**：

| 文件 | 2026-04-14 audit | 当前实际 | 增量 |
|---|---:|---:|---:|
| `virtual_intersection_poc.py` | 967 KB | **1006 KB** | +40 KB（约 +4%） |
| `stage4_step4_event_interpretation.py` | （未登记） | **118.6 KB** | 新增超阈值 |
| `stage4_geometry_utils.py` | （未登记） | **104.5 KB** | 新增超阈值 |
| `stage4_divmerge_virtual_polygon.py` | 345 KB | （已不在 src/）| 拆分/迁移未登记 |

也就是说，规则上线后的一周里：**原超阈值文件继续增长，2 个新文件直接长到超阈值，1 个旧超阈值文件被迁移而 audit 表没被更新**。任何机制都没有生效。

### 2.2 "CodeX 默认轻量编码守则"（line 25-33）与 `default-imp/SKILL.md` 重复

`AGENTS.md` 第 27-33 行的 7 条与 `.agents/skills/default-imp/SKILL.md` 的"默认实现守则"几乎一字不差。问题：

- **双源真相**：未来调整时只改一边会形成沉默冲突。
- **暗示优先级反转**：放在 `AGENTS.md` 里的版本看起来更"权威"，但 `default-imp/SKILL.md` 才是流程主真相，否则就违反了 `AGENTS.md` 自己第 6 行"标准可复用流程统一放 `.agents/skills/<skill-name>/SKILL.md`"的规定。

### 2.3 "GIS 质量要求"（line 15）

> 在涉及 GIS / 拓扑 / 空间数据处理的任务中，Agent 必须以顶级 GIS 算法工程师标准执行……

- "顶级 GIS 算法工程师标准"是不可度量的修辞，模型无法据此停机。
- 后半句"优先保证 CRS 与坐标变换正确性、拓扑一致性…"这一串才是真正可执行的清单，应当独立成条并配套"如何验证"的最小动作（哪怕是"调用 `qgis-auto-visual-check` Skill 复核"这种引用）。

### 2.4 "执行环境"与"盘符映射"（line 11-13）

- 这三条信息密度极高，但**只有用户在用 WSL 时才正确**。当前会话本身就是在 PowerShell + Windows 直接执行（终端里也确实在用 `Get-ChildItem`），第 11 行"内网与外网默认都在 WSL 下执行"会**主动误导**那些不在 WSL 的会话。
- 建议改为条件触发："如果使用 WSL，则 ……"；或者把 WSL 路径换算放到一个独立的 `docs/repository-metadata/path-conventions.md` 中，`AGENTS.md` 只保留"涉及跨盘路径必须先确认环境"这一行硬规则。

### 2.5 范围保护中的"举例"（line 23）

> 例如本轮若明确排除 `T02`，则不得修改 `modules/t02_junction_anchor/**`……

- 把具体模块名写进通用规则会**绑定特定轮次的语境**，下一轮如果保护对象换成 T03/T04，文本就会过时。
- 应抽象为"任务书显式排除的模块/目录/spec 一律视为保护区"，把示例放进 `default-imp/SKILL.md` 或一次性的 spec 工件里。

### 2.6 "详细约束"末梢引用（line 20）

> 详细约束：`docs/repository-metadata/code-boundaries-and-entrypoints.md`。

- 这是孤立的一行交叉引用，模型读取时几乎一定不会主动展开（除非显式被告知"读 X 再继续"）。
- 应改为"任何涉及 `<入口/体量/依赖>` 的改动**必须先**读 `code-boundaries-and-entrypoints.md`"，把"可选阅读"升格为"前置阅读"。

### 2.7 "字段启用约束"（line 16-17）

这两条措辞最严密，可作为**全文模板**。共同点：

- 主语 + 动词（"禁止 / 必须"）。
- 触发条件具体（"未在源事实文档中正式启用的字段"）。
- 失败回路明确（"必须先列出矛盾点，并向用户确认"）。

后续把其它规则改写时应对齐这种结构。

### 2.8 "执行入口"（line 19）

写得不错，唯一缺陷是"必须列出冲突并停止"出现在长句末尾，强信号被埋。建议把所有"列出冲突并停止"的触发条件抽出到一个集中的"硬停机触发清单"段。

### 2.9 文档语言（line 8）

合理，建议保留。可补充一句"本文件本身也用中文，不在变更轮次里改写为英文"，避免被某次 i18n 风潮误删。

### 2.10 阅读链路（line 3）

> 主入口：先读 `docs/doc-governance/README.md`；只有在需要理解仓库结构、入口边界、文件体量与目录角色时，再读 `docs/repository-metadata/README.md`。

良好。但与 `docs/doc-governance/README.md` 的"从哪里开始看"清单存在**两份排序事实**，应在 `AGENTS.md` 里只指 `docs/doc-governance/README.md` 一个入口，把详细排序留给那个文件维护，避免双源排序冲突。

## 3. 哪些适合放在 `AGENTS.md`，哪些不适合

### 3.1 适合放在 `AGENTS.md`

- **会话开始就要立刻知道、否则下一句就会犯错的规则**：
  - 阅读链路与源事实优先级
  - 项目级冲突 → 停机的触发条件
  - 范围保护（不动 outputs/、保护区、未授权不创建模块）
  - 结构债与文件体量的**前置硬约束**
  - 执行入口禁止扩张
  - 文档语言
- **可执行性强、动词强、触发条件明确**的硬规则。
- 极简的"读哪、问哪"路标。

### 3.2 不适合放在 `AGENTS.md`

- **会过期的具体事实**（例如"当前正式模块只有 T01/T02/T03/T04"、"本轮排除 T02"）→ 留给 `module-lifecycle.md` 或具体任务书。
- **路径/盘符/Windows↔WSL 换算表**等环境性信息 → 抽到 `docs/repository-metadata/path-conventions.md`（或类似）。
- **流程长描述**（CodeX 默认守则、Spec Kit 入口流程）→ 留给 `.agents/skills/<skill>/SKILL.md`。
- **修辞性高目标**（"顶级 GIS 工程师标准"）→ 改写成可度量清单或删。
- **任何引用一句话就跳走的"详细见 X"**：要么抠出关键约束放回 `AGENTS.md`，要么改成"凡涉及 X **必须先读** Y"的强引用。

## 4. 修订方向（草稿对应这些方向）

1. **加分组小标题 + 显式优先级**：让模型一眼分辨"硬停机 / 硬约束 / 软建议 / 信息"。
2. **新建"硬停机触发清单"段**：把所有"列出冲突并停止 / 必须报告并暂停"的触发集中。
3. **"代码体量"升级成三段式**：
   - 前置检查（动手前必须自检当前文件体量）。
   - 硬阈值（任意 `.py/.sh/.cmd/.ps1/.ts/.js` 单文件 ≥ 100 KB → 禁止仅追加，必须先报）。
   - 闭环（任何让某文件首次跨过 100 KB、或拆分/迁移超阈值文件的轮次，**同轮**必须更新 `code-size-audit.md`）。
4. **删除"CodeX 默认轻量编码守则"段**：改为一行强引用 `.agents/skills/default-imp/SKILL.md`。
5. **WSL/盘符外置**：`AGENTS.md` 只保留"涉及跨盘/跨环境路径前必须确认当前会话环境"。
6. **GIS 质量改为可度量清单**或并入 `architecture/10-quality-requirements.md`。
7. **"主入口阅读链路"只指 `docs/doc-governance/README.md`**：不再在 `AGENTS.md` 里维护第二份阅读顺序。
8. **任务级保护区去掉 T02 举例**：留作抽象规则。
9. **新增"完成回报最小集"**：让"已修改 / 已验证 / 待确认"成为强制最后一步，而不是建议。
10. **可选机制建议（与 AGENTS.md 解耦但应推动）**：
    - 加 pre-commit / CI 检查：阻止任何被 staged 的源码 / 脚本文件超过 100 KB 且未同时修改 `code-size-audit.md`。
    - 把 `code-size-audit.md` 的"审计日期"换成"由脚本自动刷新"的字段。

## 5. 风险与遗留问题（建议交给 GPT 二次审计时一并看）

- "GIS 质量"如果转入 `architecture/10-quality-requirements.md`，是否需要在 `AGENTS.md` 留一行"凡 GIS 任务必须先读 quality-requirements"作为强引用？
