# T08 Tool11 现状研究与决策

## 1. 现状

- T08 当前正式工具到 Tool10，采用模块 callable + `scripts/t08_tool*.py` 的入口结构。
- T08 当前命名契约只有 Tool1、Tool10 两个业务文件名特例；用户已授权 Tool11 保留复制文件原名，形成第三个特例。
- 现有 Tool10 已证明 T08 可采用临时成果、校验后替换和 summary 审计模式。
- 本需求是文件系统数据整理，不需要 GIS 解析库或新增依赖。
- 当前 `main` 工作区存在与本任务无关的 T06 未提交修改，本轮使用隔离 worktree，不触碰这些修改。

## 2. 决策

### D1 - 物理复制而非链接

采用物理复制。硬链接会共享底层内容，符号链接会引入根目录外依赖，都不符合“单独放置”和独立交付语义。

### D2 - 哈希证明逐字节一致

每个文件复制后分别读取源和目标计算 SHA-256。文件大小只用于计数和性能，不作为完整性证明。

### D3 - 整轮暂存与双根发布

主输出根和实验输出根分别在自身父目录建立同卷暂存目录。全部 Patch 复制、哈希和计数通过后才发布；覆盖时先把旧根移动到带 run token 的备份路径，任一发布失败即回滚。

### D4 - 失败 summary 位于输出根同级

summary 不放在尚未发布的输出根内。默认文件名包含 UTC 时间和短 token，并以 `_tool11.json` 结尾，因此预检失败也可以留下审计且不会覆盖旧 summary。

### D5 - 默认实验列表可显式替换

内置用户确认的 6 个 Patch 作为默认值。调用者重复传入 `--experiment-patch-id` 时，使用显式列表整体替换默认值，便于回归和后续受控复用。

### D6 - 符号链接和特殊文件硬失败

Tool11 不猜测链接应保持、解引用还是跳过。为保证输入边界和审计可解释，遇到符号链接或非普通文件/目录时整批失败。

## 3. 已解决问题

- Tool11 文件名是否必须改成 `_tool11`：否。业务复制文件保持原名，summary 保持 `_tool11`。
- 6 个 Patch 放在哪里：独立参数 `--experiment-output-root` 指定的根目录下。
- 实验数据从哪里复制：从本轮已经哈希验证的主暂存成果复制，再与主成果哈希核对。

## 4. 风险与缓解

- 大数据双重哈希增加 I/O：summary 单独记录复制校验吞吐，完整性优先于仅大小校验。
- 跨卷无法使用单个文件系统原子事务：两个根各自在同卷暂存，并使用可回滚的两阶段发布；summary 记录提交状态。
- 源数据运行中变化：复制前后比较大小与 `mtime_ns`，变化则失败并阻止发布。
- 输出路径嵌套导致递归复制：预检要求三个根两两不重叠。

## 5. 本轮未改

- 不改变 Tool1-10 算法、参数或输出。
- 不修改任何原始数据。
- 不把实验 Patch ID 提升为其它模块业务规则。
